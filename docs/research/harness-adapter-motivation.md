# Harness Adapter：动机与愿景

## 问题

### 现有生态：碎片化

Codex、Claude Code、Gemini、Devin、Cursor、Grok、Qwen Code、Kimi Code……几乎所有厂商都会推出自己的编码 Agent Harness，并尽可能想把用户锁定在自己的 Harness 和模型上。

各 Harness 相互封闭，上层工具（调度、评测、可视化、迁移）需要为每个 Harness 重复开发专属适配层。

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 调度系统  │  │ 评测系统  │  │ 可视化   │
│ (专用)   │  │ (专用)   │  │ (专用)   │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │
     ▼              ▼              ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Claude   │  │  Codex   │  │  Kimi    │
│ Adapter  │  │ Adapter  │  │ Adapter  │
└──────────┘  └──────────┘  └──────────┘
```

### 用户视角：Harness 应该是可替换的

从用户角度看，Harness 已经成为可以随意替换的组件——可以迁移、查看、调用、编排。但每个工具都独立开发一套适配代码，重复且低效。

## 解决方案

### 中立通用抽象库

一个中立的通用抽象库，专门用于兼容各个 Harness 的：

1. **历史数据**：读取各 Harness 的会话记录（消息、工具调用、token 用量）
2. **Meta 信息**：查询会话元数据（模型、时间、工作目录、状态）
3. **操作 Harness**：启动、恢复、停止会话；调用工具

### 愿景

```
一次开发适配器，全生态共享

┌─────────────────────────────────────────────────┐
│          调度系统  │  评测系统  │  可视化(MindWalk)  │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│              Harness Adapter（统一 API）          │
│  SessionStore  │  MessageStore  │  AgentStore    │
├─────────────────────────────────────────────────┤
│  PiAdapter │ CCAdapter │ KimiAdapter │ ...      │
├─────────────────────────────────────────────────┤
│  ~/.pi/  │  ~/.claude/  │  ~/.kimi-code/  │ ... │
└─────────────────────────────────────────────────┘
```

### 目标

- **消除碎片化**：一个适配器，所有上层工具共享
- **解除厂商锁定**：用户可以在 Harness 之间自由迁移
- **离线标准**：面向会话归档、Trace 回放、跨 Harness 迁移的离线标准格式（AHS: Agent History Standard）
- **通信层兼容**：兼容 ACP（Agent Communication Protocol）等标准协议

## 当前阶段

处于**调研阶段**：

1. ✅ **存储 Schema 调研**：已研究 9 个 Harness 的会话存储结构
2. 🔜 **交互接口调研**：需调研各 Harness 的 CLI 接口、ACP 支持、API 等
3. ⏳ **统一 Schema 设计**：基于调研结果设计 AHS 离线标准
4. ⏳ **适配器实现**：开发各 Harness 的 Reader/Writer 适配器
