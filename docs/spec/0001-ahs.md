# Spec-0001: AHS（Agent History Standard）格式规范

验收标准（AC）在 [ac/0001-adapter-ac.md](../ac/0001-adapter-ac.md)，适配器接口定义在 [interface/0001-harness-adapter.md](../interface/0001-harness-adapter.md)，设计取舍的决策依据见 [adr/](../adr/)。调研依据见 [research/](../research/)。

## 一、定位

AHS 是编码 Agent 会话历史的**离线交换与消费格式**，服务于调度、评测、可视化、迁移等上层工具。全局愿景见 [vision.md](../vision.md)。

明确不是：

- **不是备份格式**。无损性由源数据（`~/.claude/`、`~/.codex/` 等）提供，AHS 不复制原始记录。要归档就保留原始数据。
- **不是控制协议**。启动/恢复/停止会话走 ACP 等在线协议，AHS 只管历史与元数据。
- **不追求双向可逆**。AHS → 源格式不需要完美逆向，迁移场景只需保留主要信息以支持"继续工作"。

## 二、核心取舍

**好的设计是权衡，而非追求完全兼容。**（决策记录见 [adr/0001-lossy-projection.md](../adr/0001-lossy-projection.md)）

保留判据：*一个评审者或评测系统理解"发生了什么、花了多少代价"所需的信息保留，其余丢弃。*

### 保留

- 对话内容：用户消息、助手消息、thinking（明文）、工具调用 + 结果
- 大输出处理：任何文本块或工具结果超过 64 KiB 时外置为 blob 引用（content-addressed），不内联（见"磁盘布局与 blob 外置"）
- 粗粒度 usage：input / output / cached / reasoning token 总量 + cost + durationMs
- 关系结构：session 间的 lineage（历史血统）与 invocation（调用来源）边
- 关键状态事件最小集：turn 边界、compaction 标记、模型切换、目标判定——它们决定"这段对话的上下文是什么"，属于语义而非过程

### 丢弃

- 流式与加速信息：chunk 级流式日志（Grok `updates.jsonl`）、ITL/TTFT 时序、step 边界（压平进 turn）
- 过程与机制：queue-operation、rate_limits、ephemeral cache 分级（1h/5m）、`server_tool_use`、`inference_geo`
- 冗余表示：Codex `response_item`/`event_msg`/顶层三重复、Grok 四层表示——导出时去重，取规范表示
- 双/多时间戳格式：统一 ISO 8601，不保留原始格式
- 加密推理 blob：Codex `encrypted_content` 无法解密，丢弃；Grok 的明文 `summary[]` 保留为 thinking
- harness 特有遥测：Qwen `attribution_snapshot` / `file_history_snapshot`、Cursor per-commit 代码归因等
- 细粒度 usage 之外的计费细目（保留总量，丢弃分层）

### 有意丢弃 = 规范的一部分

规范必须明确列出丢弃类别，让消费者一眼知道边界。**不留 `raw` / `extra` 逃生舱**——逃生舱把"要不要兼容"的决策推给每个适配器作者，导致各家填法不一致，等于没有标准。

### 源端不可得的信息

以下不是 AHS 的丢失，是源端就没有：Codex 推理内容（加密且无 summary）、Cursor 消息正文（云端）、OpenCode 全部会话数据、Devin 桌面端会话（云端）。对应适配器返回 partial / null，并在能力声明中标注。

## 三、概念模型

存储层只有两类实体：**Session** 和 **Relation**。一切视图（turn、task、主链）都是投影；存储只有线性 session 和边。（决策记录见 [adr/0002-session-relation-model.md](../adr/0002-session-relation-model.md) 与 [adr/0005-linear-sessions.md](../adr/0005-linear-sessions.md)）

### Session（存储视角）

- **Session 是一个 agent profile（声明）的一次实例化**。profile 可以是显式的（Claude `--agents`、Grok `--agent-profile`、Devin `--agent-config`）或默认的（Codex、Kimi 等无 profile 机制）。
- 一个 agent（= profile 声明）可以有多个 session。
- **Session 的历史一律线性**：record 只以 `seq` 排序，无 parentId、无内部分叉。一切分叉（rewind、编辑重发、重试、竞争尝试）一律产出新 session。
- Subagent 不是特殊对象，**就是另一个 session**——在历史层它由父 session 中一次普通的 tool_call/tool_result 调起。

### Relation：两个正交维度

**lineage（历史维）——"我的历史从哪来"**：

```typescript
lineage?: { type: "forked_from" | "sibling_attempt", sessionId: string, atRecordId?: string }
```

- fork session 只存分叉后的**后缀**，共享前缀沿 lineage 回溯拼接。
- 判据（数据判据，与发起者无关）：`forked_from` = 锚在 agent 侧 record 之后（分叉后以**新 user 输入**继续，含编辑重发）；`sibling_attempt` = 锚在 `user_message` 之后（对**同一个 prompt** 重新作答，互为竞争）。`atRecordId` 为 null 表示从起点重试。

**invocation（调用维）——"谁调用/创建了我"**：

```typescript
invocation?: { sessionId: string, atRecordId?: string }  // atRecordId = 父中 spawning tool_call 的 recordId
```

- 正链在历史层：`tool_result.sessionId`（可选）记录该调用产生的子 session（tool_call 时 id 未存在，不可能携带）。
- fork 的 invocation 沿 lineage **传递继承**，不显式复制。

**关系存储 = 纯派生索引**：双向边表、组指针、传递闭包、聚合视图全部由 sessions + manifests 派生，非 authored 数据。

### Task（用户视角，派生概念）

**Task = lineage 组 + HEAD 指针**，是用户视角的"一个会话"（一次业务任务的全部尝试 + 当前位置），类比 git repo（不可变 commit 链 + 可变 HEAD）。

- lineage 组：lineage 边向上走到同一根的全部 session（计算得出，无需存储）。
- **HEAD 指针** `{ groupId, mainSessionId }`：组级可变指针，存于派生层；源端有胜者标记（Devin `main_chain_id`）则直取，否则取组内最近活跃 session。winner 易主只移动指针，session 不变。
- Task 不是存储实体。Codex 跨文件线程、Devin slug 会话均为 Task 的实例。

## 四、数据模型

### 模型骨架

```
Manifest（会话级，一份）        Records（record 级，JSONL，append-only，线性）
├─ sessionId                   ├─ recordId
├─ harness + harnessVersion    ├─ seq（时序号，唯一结构字段）
├─ ahsVersion                  ├─ timestamp（ISO 8601）
├─ profile?（声明或默认）       ├─ type + role + content[]
├─ cwd / workspaceRoots[]      ├─ model?（覆盖，应对中途切换）
├─ git {branch, commit, url}   ├─ usage?
├─ model / provider            └─ ...
├─ title + titleOrigin
├─ lineage?（历史维回链）
├─ invocation?（调用维回链）
├─ acpBinding {agentId, sessionId}
└─ stats（本 session 独占聚合）
```

统计口径：一个任务的完整代价 = 主 session + 沿 invocation 递归（含经 lineage 传递到达的 fork）聚合。fork 只存后缀，无共享前缀双计。`stats` 只存本 session 独占值，跨 session 聚合交给消费方沿关系图走。

### Record 类型（最小集）

Content 类：

- `user_message`（**纯人类输入**；harness 注入的伪装用户消息不落这里）
- `assistant_message`（content blocks：text / thinking / image；thinking 是 block 而非独立 record）
- `harness_message`（harness 注入的无结构消息：ralph 重提示、cron 提醒、background 完成通知、system reminder。内容模型与 user_message 相同，仅 provenance 不同——来源用一等类型表达，不用字段标签。适配器只映射源端显式合成标记（如 Grok `synthetic_reason`、Kimi `origin.kind`）；源端无标注时落入 user_message，属源端不可得）
- `tool_call`（保留原始 tool name，kind 分类为可选派生字段；`status?: "completed" | "failed" | "interrupted"`——源会话中断导致无配对 tool_result 时标 `interrupted`，**不合成假 result**；同一 toolCallId 出现多个 result（重发/分支）时保留文件序第一条）
- `tool_result`（关联 toolCallId；大输出走 blob 引用；`sessionId?`——该调用产生子 session 时的正链，仅此类调用携带）

状态类（最小集）：

- `turn_boundary`（turn 开始/结束，含中断结束如 Codex `turn_aborted`。源端有生命周期事件时发射，没有时省略——适配器不得合成；统计口径由 Manifest `stats.turnCount` 兜底）
- `model_change`
- `compaction`
- `goal_update`（目标判定的结构化记录：`{ goalId?, status: "pending" | "met" | "unmet", reason? }`。目标**创建**走工具调用（Kimi/Claude 的 `CreateGoal`）天然是 tool_call，不在此列；本类型只承载 harness 控制面的判定事件——Claude `attachment.goal_status`、Codex `thread_goal_updated`。映射约定：Claude `sentinel: true`（监控开始）→ `pending`，`met: true` → `met`，判定未达成 → `unmet`；源端无 goalId 时省略，多次判定靠 seq 关联）

**provenance 原则**：record 的"谁写的"完全由类型联合表达（一等结构优于标签字段）。human → `user_message`；model → `assistant_message` / `tool_call`；environment → `tool_result`；harness 策略 → `harness_message` / `goal_update`。

### 磁盘布局与 blob 外置

一个 AHS 归档由一个 sessions 根目录组织，每个 session 一个目录：

```
<sessions-dir>/<sessionId>/
├── manifest.json        # Manifest（会话级元数据与聚合统计）
├── records.jsonl        # 全部 record，一行一条，按 seq 升序
└── blobs/
    └── sha256-<hash>    # 被 BlobRef 引用的大内容
```

派生层（可选）：归档根下可有 `relations.jsonl`（双向边表 + 组指针），由 sessions + manifests 派生，供父→子导航、Task 视图与聚合；删除后可随时重建。

**blob 外置规则**：

- 任何文本块或 tool_result 内容超过 **64 KiB** 时外置，record 内留 `BlobRef { type: "blob_ref", sha256, mediaType, byteLength, preview }`
- blob 文件以内容的 SHA-256 命名（content-addressed）：天然去重、完整性可校验、不可变
- `preview` 为内容前 256 字符，消费方不解 blob 即可展示摘要
- 阈值与 obsidian-harness-frontend 的生产验证设计一致；Claude `tool-results/` 等源端外置文件在归档时按同一规则内联或转 blob

### Usage 结构

```typescript
interface Usage {
  inputTokens?: number
  outputTokens?: number
  cacheReadTokens?: number
  cacheWriteTokens?: number
  reasoningTokens?: number
  cost?: { amount: number; currency: string }  // 仅少数 harness 有
  durationMs?: number
}
```

record 级可选；Manifest.stats 聚合本 session 总量。

### Manifest 字段详表

> 字段可得性：部分 harness 源端不提供某些字段——允许为空或 `"unknown"`（如 Kimi 的 cwd 是 hash 不可还原、CLI 版本不存在）。适配器不得编造缺失值。

| 字段 | 说明 | 来源依据 |
|------|------|---------|
| `sessionId` | 稳定且与 mainness 解耦；原样保留 ULID / UUID / slug 形态，fork 加确定性后缀 | 各家 ID 类型不一致，不强制 UUID |
| `harness` / `harnessVersion` | harness 标识 + CLI 版本 | |
| `ahsVersion` | AHS 规范版本 | |
| `profile` | agent profile 声明；无 profile 机制时为默认/省略 | Claude `--agents`、Grok `--agent-profile` |
| `cwd` / `workspaceRoots[]` | 工作目录 | Codex 有多 root |
| `git` | branch / commit / repoUrl | Codex 最全 |
| `model` / `provider` | 主模型；中途切换走 record 级覆盖 | Pi 有 model_change |
| `title` / `titleOrigin` | generated / custom | Kimi、Grok、Devin、Cursor 有 |
| `lineage` | 历史维回链（forked_from / sibling_attempt + atRecordId） | ADR-0005 |
| `invocation` | 调用维回链（谁创建了我 + atRecordId） | ADR-0005 |
| `acpBinding` | {agentId, sessionId}，接回原生会话续接 | 借自 obsidian-harness-frontend |
| `stats` | 本 session 独占聚合：totalUsage、turnCount、durationMs | Qwen usage_record、Grok signals |

## 五、源模型归一化

五种源存储模型统一到"线性 session + 两维关系"：

| 源模型 | Harness | 归一化方式 |
|--------|---------|-----------|
| Tree | Pi, Qwen | 线性直译；源内分叉拆为 fork session；Qwen 内联 managed subagent（auto-memory-extractor）只有遥测无内容，不产生子 session，其按 `source` 区分的聚合 usage 并入 Manifest.stats 记账 |
| Graph + sidechain | Claude | 主文件线性化（分叉拆 fork session，last leaf 为主线启发式）；subagent 文件为独立 session（invocation 回链锚 toolUseId 对应 record；正链写入父中 Task tool_result.sessionId） |
| Forest | Devin | 每个 root 一个 session；非首 root 为 fork（共享前缀锚点，锚于 user_message 者为 sibling_attempt）；`main_chain_id` → 组 HEAD 指针 |
| Temporal stream | Kimi, Codex, Grok | 线性直译（seq 保序）；Kimi 每个 wire.jsonl 一个 session（子 wire invocation 回链无 atRecordId）；Codex thread_spawn → invocation（正链经 sub_agent_activity 对账）、lineage 文件祖先 → forked_from |

## 六、与 obsidian-harness-frontend 的关系

AHS 是其 TranscriptItem schema v2 的**超集**。AHS 多出：usage、两维关系结构（lineage / invocation）、状态事件（turn_boundary / model_change / compaction / goal_update）、Manifest 聚合统计。

AHS 采纳其经过实战检验的设计：blob 外置（64 KiB 阈值、content-addressed）、uuid5 幂等 ID、acpBinding 续接、forkedFrom 关系。

主要差异：AHS 是 **record 级**（turn 是投影结果，不是存储格式）；obsidian 的 turn 级存储面向渲染，是 AHS 的一种上层投影。

## 八、已决事项与规范演进

- blob 布局与阈值 → 见"磁盘布局与 blob 外置"（64 KiB、content-addressed、preview 256 字符）
- `turn_boundary` 形态 → 保留独立 record 类型，源端有才发射，见"Record 类型"
- Qwen managed subagent → 不产生子 session，usage 并入 stats，见"源模型归一化"
- Codex 跨文件线程分组 → 由 Task 概念解决（lineage 组 + HEAD 指针），见"概念模型"
- 规范存放位置 → 暂留本仓库与实现同仓版本化；**迁出触发条件**：出现第二个消费方/实现，或发布 AHS v1.0 标准时独立成仓

规范的后续演进：本文件 status 流转（draft → proposed → active）遵循 devloop 契约冻结规则；active 后不可原地修改，修订须退回 DESIGN 走 ADR 流程。

## 九、术语表

| 术语 | 定义 |
|------|------|
| AHS | Agent History Standard，本规范定义的离线交换格式 |
| Harness | 编码 Agent 的运行外壳（ReAct 循环、工具、Hooks），如 Claude Code、Codex |
| Session | 一个 agent profile 的一次实例化；历史一律线性（存储视角） |
| Record | session 历史中的单条记录（JSONL，append-only，以 seq 排序） |
| lineage | session 的历史维回链（forked_from / sibling_attempt + atRecordId） |
| invocation | session 的调用维回链（谁创建了我）；正链为 tool_result.sessionId |
| Task | 用户视角的"一个会话"：lineage 组 + HEAD 指针（派生概念，非存储实体） |
| HEAD 指针 | 组级可变指针 { groupId, mainSessionId }，存于派生层，类比 git HEAD |
| Manifest | session 级元数据与聚合统计，一个 session 一份 |
